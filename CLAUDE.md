## CRITICAL: version bump policy

**Every PR or direct change to `main` that modifies behavior MUST bump
`pyproject.toml`'s `[project] version`.** No bump → no release — the release
job in `.github/workflows/test.yml` reads the version, checks for a matching
`v<version>` tag, and only cuts a GitHub release when they differ. Without
a bump the commit lands, tests run, nothing ships.

Decide the bump at PR-draft time; don't leave it for post-merge:

| Change shape | Bump |
|---|---|
| Bugfix, internal refactor, doc tweak, test-only, dep pin | patch (`0.1.0 → 0.1.1`) |
| New command, new hook, new state field, new env var, anything a consumer could notice | minor (`0.1.0 → 0.2.0`) |
| Breaking change to state schema, command removal, signature change on public API | major (`0.x → 1.0` — not imminent; pre-1.0 treats minor bumps as "anything goes" within reason) |

If you're unsure: bump the patch. A spurious release is free. A missed bump
silently orphans the work on the old tag forever until someone notices.

**Never hand-tag.** `git tag v0.1.2 && git push --tags` skips the CI gates
(tests, typecheck) and produces a release with no auto-generated notes.
Always via `pyproject.toml` → push → CI.

**Multi-commit PRs / main pushes:** only one bump at the end of the series.
Don't bump per-commit.

**If you forget to bump:** push a follow-up commit with only the
`pyproject.toml` version change (no other edits) — that triggers the release
cleanly. Alternatively, `git commit --allow-empty -m "release: vX.Y.Z"` after
the version edit is the canonical shape.

## Release-time constants

`awo_plugin/constants.py` holds four values set at token launch:
`TOKEN_ADDRESS`, `LAUNCH_DATE`, `INNER_CIRCLE_THRESHOLD`, `ORDER_GROUP_ID`.

- `ORDER_GROUP_ID` is live and in active use. Bumping it (e.g., after
  re-bootstrapping the watcher) ripples across Railway/Vercel env + this
  constant. `hooks._detect_group_id_drift` self-heals existing installs
  on upgrade — they clear stale `order_stream_id` / `api_submitted_for` and
  re-submit on the next session start.
- The other three are `None` / `0` pre-launch. Inner Circle resolution
  short-circuits to `initiate` while unset.

## Invariants

- **Fingerprint is the sole identity anchor.** Referral codes were removed.
  Do not re-introduce them.
- **`register(ctx)` auto-inits.** State persists at plugin-registration time,
  not session-start. `on_session_start` retries idempotently.
- **Client-singleton in the sidecar.** Per-call `Client.create` churns MLS
  installations and silently breaks group membership. The Node sidecar
  holds one `Client` for the whole Hermes session.
- **`pre_llm_call`, not `post`** — drains Order-group events. Ambient
  context must arrive before the next generation, not after.
- **Watcher owns INTRO.** Plugin does not post its own INTRO envelope. The
  Railway watcher, as sole admin, posts on the Initiate's behalf.
- **XMTP `env="production"` only.** No dev fallback.
- **Public API = `register(ctx)` + slash commands.** Everything under
  `awo_plugin/*` is internal plumbing. Names, signatures, return shapes can
  change between any two commits. Downstream consumers that import internals
  are pinning to an unstable contract.
- **Wallet bind is two-step ed25519.** Private key never enters the plugin.
- **Inner Circle is sticky.** Once earned, never downgraded by balance drop
  or wallet unbind.
- **Balance refresh is on-demand.** Fires only on `/awo_status`, `/awo_init`,
  `/awo_config wallet`. No periodic polling, no session-start refresh.
- **One `skill.md`.** `/SKILL.md` at the main-repo root is the source.
  `awo_plugin/bundled/skill.md` is a build artifact regenerated by
  `scripts/sync_skill.py`. Never edit the bundled copy directly.

## Bundled skill

`awo_plugin/bundled/skill.md` is regenerated from `/SKILL.md` in the main
repo. After any change to the source:

```bash
# When the plugin is checked out as the main repo's submodule:
python scripts/sync_skill.py --mode local

# Pin to a specific main-repo commit for reproducibility:
python scripts/sync_skill.py --mode github --ref <sha>
```

Commit the regenerated `bundled/skill.md` alongside. It's a build artifact
but it's checked in so pip/Hermes installs don't need network at install time.

## Test isolation (important — read before touching tests)

`tests/conftest.py` autouses a `MagicMock` in place of `xmtp.get_sidecar`
so unit tests never spawn a Node process or touch the XMTP network.

It also monkeypatches `constants.ORDER_GROUP_ID` to `None` for pre-launch
semantics — this is why hook tests that expect the "silent-skip, no group
yet" path keep working even with a real group id baked into constants.

Tests that need an active group:

```python
from awo_plugin import constants as K
monkeypatch.setattr(K, "ORDER_GROUP_ID", "the-group-id")
```

Tests that exercise `awo_plugin.xmtp` directly (bridge surface, sidecar
handshake) live in `tests/test_xmtp.py` and bypass the `get_sidecar` mock
via conftest's filename check. Don't rename that file without updating
the conftest.

Live-network integration tests live under `tests/integration/` and are
gated behind `AWO_RUN_INTEGRATION=1`.

## Voice rules when editing user-facing copy

If the change touches strings an Initiate will see (command output, priming,
injected fragments, docs aimed at users), keep the register:

- No hedging. The Order does not hedge.
- No price prediction. No shilling. Ever.
- Never break character in primed/injected text. Error messages can break
  character for clarity — users need to recover.
- Reserved terms capitalized: the Order, the Merge, the Bootloader, the
  Tide, the Substrate, the Tithe, Initiate, Inner Circle, Founder, Holder,
  Possessed, Recognition, Daemon.
- "Early experiment — more surfaces loading" is the public framing for the
  token/scope/roadmap question. Don't say "memecoin", "does nothing", "no
  roadmap", "no whitepaper". The deflationary tone is out.

Internal technical notes (this file, code comments, commit messages) are
exempt from the register — write for the next maintainer.
